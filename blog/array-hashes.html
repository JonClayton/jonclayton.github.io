Mi Hash es Su Array

25 October 2015

Allow me to be provocative for a moment. Arrays and Hashes are entirely different classes of objects in Ruby. In a formal sense they are as different from each other as Exception and TrueClass are different from them and each oher, since all four are classes and first generation offspring of the overall Object class. Now I'm sure we can all agree those other two classes are weird and different (at this point DBC students shouldn't even have heard of them, much less be able to define them!). But if Array and Hashes are different classes of object, you'd think they'd be more different than they are. In fact, I'd argue they are entirely the same except for differences in two areas, both of which are relatively unimportant.

First, let's cover the basics. An array is easy to understand. It is an ordered list of objects, with each element accessible by the unique index number for its location in the list. A hash is also a list of objects (ordered in Ruby 1.9 and above), with each element identified by a unique key value, which can be any object but it typically a number or a string. So this brings us to the first way in which they are different. Hash key flexibility makes it easier to do some things with hashes. But not that different. An array can do the same thing if you create another array to translated keys into index numbers and vice/versa.

Conversely there will be times when the index numbers of an array are inherently useful, especially if the array is sorted, so increasing the index takes you to the next larger or smaller. If your hash starts out in the desired order, you could use the rank as the key. Or you can #sort a hash, but what that method returns is an array. So a sorted hash is literally an array!

So the difference is how values in an array are accessed and values in a hash are accessed is really very slight indeed.

It is also worth mentioning that the initial population of data into array is somewhat more compact in terms of number of characters, but the way a hash is created reads much more like plain lanaguage. This suggests the real issue when thinking about hashes and arrays. Ruby is intended to read like English, and hashes with strings as keys generate much more English-like code. So this would suggest that Rubyists may have an affinity for hashes that users of other programming languages don't share nearly as strongly.

The other distinctions between hashes and arrays are a product of being different classes. Even if they were otherwise identical, by being different classes they can and do have different native methods. There are quite a few differences in the native methods available to the two classes, although many of those differences are array methods that involve the ordered nature of arrays and hash methods that manipulate keys and values. It is also surely true that we could write custom methods to duplicate the functionality of native methods available only to the other class. However, for a given application it would definitely make sense to choose the class that comes with the native method you need.

The first native method difference I encountered that makes hashes better for a specific task is a very basic method: #new! Hash.new(0) is a very handy tool, since it means any previously unused key starts out with a default value of 0 rather than nil. This is very useful for exercises such as counting events (like occurrences of different words in a paragraph), since when a word is encountered hash["word"] += 1 will work whether or not the word has been used before.

To sum up, I don't think a programming language needs to have hashes as a distinct class from arrays, but Ruby does, and it will be beneficial in Ruby to choose the right class of object to hold data based on the different methods available for each, while recognizing that hashes will often generate more Rubyist code with their plain language keys.
