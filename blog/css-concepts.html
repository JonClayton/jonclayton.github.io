I play softball. I love playing softball. I'm pretty good at it too. I can play anywhere in the field and do a good job, even if I am more confortable in some positions than in others. This makes me a "utility player." On youth softball teams this is a euphemism for "not good enough to start," but in our softball league it's a good thing. No matter who is missing for the game, I can fill in there and we're still tough on defense. So I'm glad I can play any position.

About now you're thinking, "What does this have to do with a technical blog?" That's if I'm lucky. If I'm unluckly, you think I've lost my mind or you've clicked away to Paris Hilton's blog. Yes, there is such a thing and if you want to spend a few minutes you'll never get back, here's the link

But getting back to my point, and I do have one, it is a good thing for me to not really care about my position in softball. It is a bad thing indeed to not be in the right position in CSS. This week in Dev Bootcamp we spent time on CSS and positioning turned out to be a real challenge. I've read through other people's weekly reflections (they thought they were writing to their small Accountability Group but it turned out to be the whole cohort--but that's nother issue). I'm not the only one who didn't get it right away. So to help me make sure I get it, and perhaps to help someone else (you?) understand positioning and a few other topics, I'm going to explain them here.

Positioning is how elements are placed on the page and/or screen when you are looking at a webpage. Without positioning, elements on the page would just follow one after another, looking much like the second generation of websites. First generation being text only, second generation having some photos as elements, but every element started from the left of the screen below the previous element. So here's a summary of the four basic position properties:

  • Static (default): follows the flow of the page, so position is based on that of the previous element without overlapping. If no other instructions, this means below the previous element. No top/bottom or left/right properties allowed, so these elements will always land where you expect them to, which is both nice and VERY predictable.
  • Relative: top/bottom and left/right properties create that much space on that side of the element from where it would have been in the Static location. So Relative without non-zero top/bottom/left/right properties will be in the same place as Static would be.
  • Fixed: element goes in the position on the screen defined by top/bottom and left/right, and remains there regardless of scrolling. The direction properties describe the distance between the edge of the element and the edge of the screen.
  • Absolute: element goes in a position relative to the previous element, so it is like the fixed positioning, except the prior object takes the place of the screen. Note that the direction properties, if positive, are describing locations inside the edges of the previous object.
